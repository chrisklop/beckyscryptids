# Playbook v0.3 Phase 1 — Cycles + Managers + AdCap-style Card Layout

> Closes the gameplay-feel gap from v0.2 playtest: "I'm at 6.7K/s, nothing is happening, just whack-a-mole on Recommended." AdCap's signature mechanics (per-building production cycles, manager-hire-as-purchase, animated cycle bars) restored. Math foundation untouched — discrete cycles average to the same rate-per-second as our continuous tick.

**Out of scope for Phase 1:** upgrades, achievements, audio (those ship as v0.3 Phase 2).

**Math invariants:** all 12 reachability tests must keep passing. The cycle model is mathematically equivalent in steady state.

---

## 1. Production model: discrete cycles instead of continuous tick

### 1.1 Schema additions to `GeneratorTier`

```ts
cycle_seconds: z.number().positive().default(1.0),
manager_cost: z.number().nonnegative().default(0),
manager_name: z.string().default('Manager'),
```

### 1.2 Era 1 per-generator values

| Generator | cycle_seconds | manager_cost | manager_name |
|---|---|---|---|
| spread-rumor (T1) | 1.0 | 100 | Sycophant Manager |
| forge-naru-tablet (T2) | 3.0 | 5,000 | Scribe Manager |
| smear-rival (T3) | 6.0 | 50,000 | Pamphleteer Manager |
| hire-sykophant (T3) | 8.0 | 100,000 | Sykophant Boss |
| bronze-coin-mint (T4) | 15.0 | 2,000,000 | Mint Master |

Higher tier = slower cycle = bigger payout per cycle. Cycle time roughly doubles per tier.

### 1.3 Payout-per-cycle formula

```
payout_per_cycle = base_production × cycle_seconds × owned × milestoneMult × globalMultiplier
```

This makes rate-per-second equal to `base_production × owned × milestoneMult × globalMultiplier` — **identical to the current tick model.** All reachability invariants pass without recalibration.

### 1.4 Per-generator runtime state

`cycleProgress: number` (0..1) per generator, added to `state.cycleProgress: Record<genId, number>`.

### 1.5 Tick logic

Every 100ms tick (`dt = 0.1s`):

```ts
for each gen the player owns >= 1:
  if gen.managerHired OR gen has been manually triggered this cycle:
    cycleProgress[gen.id] += dt / cycle_seconds
    if cycleProgress[gen.id] >= 1:
      payout = payout_per_cycle
      state.rumor += payout
      state.lifetimeRumor += payout
      cycleProgress[gen.id] = 0
      fire popper animation '+payout' on the card
```

For non-manager-hired generators: cycle doesn't auto-advance. Player must tap the card to advance one cycle (one tap = one cycle's worth of payout once bar fills).

**For Tier 1 click-driven** (special case to preserve the AdCap "tap the lemonade stand" bootstrap):
- Tap always adds +1 Rumor (free click) — KEEP
- Tap also buys one if affordable — KEEP
- Tap also advances Tier 1's cycle bar by `cycle_seconds × 0.5` per tap (so ~2 taps = one cycle payout)
- Once Sycophant Manager is hired (100 Rumor), cycle auto-runs and player no longer needs to tap for income

### 1.6 Removing the old auto-unlock mechanic

`auto_unlock_at` field in the schema becomes vestigial — managers replace it. We keep the field in the schema for backward compat but stop checking it in `generatorProduction`. The condition becomes:

```ts
if click-driven AND !managerHired AND !manuallyTriggeredCycle:
  return 0
```

In practice, for v0.3 we'll set Tier 1 manager_cost = 100 (cheap, hits in first 2-3 minutes) so the "Sycophant hired" moment fires fast.

## 2. Manager-hire UX

### 2.1 Schema + state

- Schema: `manager_cost`, `manager_name` per generator (above)
- State: `state.managersHired: Set<string>` of generator ids

### 2.2 UI affordance

Each generator card gets a "Hire {manager_name}" button (small, below the cycle bar) when:
- The generator has `owned >= 1`
- `managersHired` doesn't include this gen
- `manager_cost > 0` (Tier 1 has 100; everything has a cost)

Button shows `Hire {manager_name} — {formatCost(manager_cost)}`. Tapping spends Rumor and adds gen.id to `managersHired`. Fires `manager-hired:{era}:{gen_id}` toast.

### 2.3 Visual difference between manual vs auto

- Manual: cycle bar visually FROZEN at its current value when not being tapped. Player must tap to advance.
- Auto (post-hire): cycle bar smoothly fills + pops in a loop.

A small icon next to the manager name indicates: `🔓` (locked, must tap) → `✓ {manager_name}` (auto).

## 4. Number poppers

Each cycle payout fires a `+formatPayout(amount)` label that floats up off the cycle bar over 800ms and fades. Reuses the existing `.mile-pop-label` animation pattern from v0.2.

## 5. Save migration v2 → v3

```ts
interface SaveStateV3 extends SaveStateV2 {
  cycle_progress: Record<string, number>; // gen_id → 0..1
  managers_hired: string[];               // gen_ids of hired managers
}
```

V2 → V3 defaults: empty maps. Existing players continue smoothly; Tier 1 will require buying its manager (cheap at 100 Rumor) to restore auto-income.

## 6. State refactor

`src/ui/state.ts`:
- New reactive fields: `cycleProgress: Record<string, number>`, `managersHired: Set<string>`
- `buyGenerator()` unchanged (still buys generators)
- New action: `hireManager(genId: string): boolean` — spend rumor, add to set, fire toast
- New action: `manualCycleTap(genId: string)` — for Tier 1 click-driven before manager: advances cycle by `cycle_seconds × 0.5`. For other tiers before manager: same advance.
- Tick loop replaced: instead of `state.rumor += rate × dt`, iterate over generators and advance cycle progress, fire payouts on cycle completion.

## 7. Test additions

- `tests/game/cycle.test.ts` — verifies cycle math: 1 cycle of a single-owned gen produces `base_prod × cycle_seconds × global` rumor; rate averages to `base_prod × global` per second.
- `tests/game/manager.test.ts` — `hireManager` requires affordable cost, persists in state, only triggers once.
- Reachability invariants in `tests/economy/` — must still pass without changes (math equivalence).
- `tests/game/save-migration.test.ts` — v2 → v3 adds `cycle_progress: {}` and `managers_hired: []` defaults.

## 8. Simulator update

The simulator must model cycles too, otherwise reachability timings change.

Two options:
- (a) Keep simulator's continuous-tick model and accept tiny precision differences.
- (b) Update simulator to cycle-discretization.

Pick **(a)** — math is steady-state equivalent, and the simulator's purpose is verifying long-horizon timings (~30-60 min), where discretization noise is negligible.

## 9. UI behavior summary

| Action | Effect |
|---|---|
| Tap a card's main body | If T1: +1 Rumor, +cycle progress, +auto-buy if affordable. Other tiers (manager not hired): +cycle progress. After manager: no effect, cycle auto-runs. |
| Tap "Buy ×N" button on card | Purchase N more of generator |
| Tap "Hire Manager" button | Spend manager_cost, hire manager, cycle starts auto-running |
| Cycle bar reaches 100% | Payout = `owned × base × cycle_seconds × milestones × global`; bar resets to 0; popper animation fires |

## 10. Success criteria

After Phase 1 ships:
- Player sees cycle bars filling continuously on every owned generator (game feels alive)
- Manager-hire moment is a real decision ("save up for Sycophant Manager, or buy more Tier 1 first?")
- Number poppers visible on every cycle payout
- AdCap-style card layout in place (icon, owned count, cycle bar, action row)
- All 1405 + new tests pass
- Reachability invariants still 12/12 (Era 1 first prestige at ~42 min unchanged)
